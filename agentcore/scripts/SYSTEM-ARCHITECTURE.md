## 1. Application Overview

**Autonomous Agency** is a multi-tenant AI-powered recruitment and sales automation platform. It orchestrates a pipeline of specialized AI agents that autonomously:

- **Discover** candidates or prospects via web search
- **Scrape & parse** LinkedIn profiles, CVs, and company pages
- **Enrich** contacts from 6вҖ“8 parallel sources (GitHub, Twitter, Stack Overflow, company sites, etc.)
- **Score** each contact against job requirements or ideal customer profiles
- **Send** personalized, multi-step email campaigns generated by Claude AI
- **Listen** for replies via IMAP/POP3 polling
- **Classify** replies (interested, objection, out-of-office, bounce, unsubscribe, etc.)
- **Act** on positive replies (schedule interviews, send candidate reports)
- **Thread** all email conversations into a CRM with automatic deal progression

The platform supports two primary use cases:
- **Recruitment** вҖ” Automated talent sourcing and candidate outreach
- **Sales** вҖ” Automated lead generation and decision-maker outreach

## 2. Tech Stack

### Backend (`agentcore/`)

| Layer | Technology | Purpose |
|-------|-----------|---------|
| Framework | Fastify 5.2 | HTTP server, WebSocket, route handling |
| ORM | Drizzle ORM 0.38 | Type-safe PostgreSQL queries and migrations |
| Database | PostgreSQL 16 | Primary data store with RLS |
| Connection Pool | PgBouncer | Transaction-mode pooling (1000 clients вҶ’ 20 pool) |
| Cache & Queues | Redis 7 + BullMQ 5.30 | Job queues, caching (7-day page cache, 30-day email cache), pub/sub |
| AI вҖ” Generation | Claude (Anthropic SDK) | Email copywriting, auto-responses |
| AI вҖ” Extraction | Together AI (DeepSeek-V3) | Classification, scoring, data extraction, profile synthesis |
| Web Search | SearXNG | Meta-search engine for candidate/company discovery |
| Web Scraping | Crawl4AI | Headless browser scraping вҶ’ markdown |
| Email Sending | Nodemailer (SMTP) | Outbound email with tracking pixels |
| Email Receiving | ImapFlow / node-pop3 | IMAP/POP3 inbox polling |
| Auth | JWT + bcrypt | Access tokens (15 min) + refresh tokens (7 days in Redis) |
| Validation | Zod | Runtime schema validation for env vars and request bodies |
| Process Manager | PM2 | Production process management with memory limits |

### Frontend (`dashboard/`)

| Layer | Technology | Purpose |
|-------|-----------|---------|
| Framework | Next.js 15.1 (App Router) | React server/client rendering |
| UI | React 19 + Tailwind CSS 3.4 | Component rendering and styling |
| Components | Radix UI | Accessible dialog, toast, label primitives |
| Icons | Lucide React | Consistent icon set |
| State (Server) | TanStack React Query 5.62 | API data fetching, caching, and invalidation |
| State (Client) | Zustand 5.0 | Auth store (sessionStorage) + real-time event store |
| Charts | Recharts 3.7 | Dashboard analytics visualizations |
| Drag & Drop | @dnd-kit | CRM kanban board |
| File Upload | React Dropzone | Document upload (CV, job specs) |
| Real-time | WebSocket (custom manager) | Live agent events, auto-reconnect with backoff |

## 5. Database Schema

25 tables organized by domain, all with UUID primary keys and UTC timestamps.

### Authentication & Tenancy

| Table | Key Columns | Purpose |
|-------|------------|---------|
| `tenants` | name, slug, plan (free/starter/pro/enterprise), productType (recruitment/sales/both), settings | Multi-tenant isolation root |
| `users` | tenantId вҶ’ tenants, email, passwordHash, role (owner/admin/member/viewer) | User accounts |

### Agent System

| Table | Key Columns | Purpose |
|-------|------------|---------|
| `masterAgents` | tenantId, name, mission, useCase, status (idle/running/paused/error), config (JSON) | Pipeline orchestrators |
| `agentConfigs` | masterAgentId, agentType (10 types), systemPrompt, tools, parameters, isEnabled | Per-agent configuration |
| `agentTasks` | masterAgentId, agentType, status, priority, input/output (JSON), retryCount | Job execution history |
| `agentMemory` | masterAgentId, agentType, memoryType (short/medium/long_term), key, value, expiresAt | Agent state persistence |

### Contacts & Companies

| Table | Key Columns | Purpose |
|-------|------------|---------|
| `contacts` | masterAgentId, firstName, lastName, email, emailVerified, linkedinUrl, title, companyId, skills (JSON), experience (JSON), score, scoreDetails, source, status | Candidates / prospects |
| `companies` | masterAgentId, name, domain, industry, size, techStack (JSON), funding, description | Target companies |
| `documents` | masterAgentId, contactId, type (job_spec/cv/linkedin_profile/...), extractedData, rawText | Uploaded / scraped documents |

### Campaigns & Email

| Table | Key Columns | Purpose |
|-------|------------|---------|
| `campaigns` | masterAgentId, name, type (email/linkedin/multi_channel), status, stats (JSON: sent/opened/replied/meetingsBooked) | Outreach campaigns |

### Email Infrastructure

|-------|------------|---------|

| `crmActivities` | contactId, dealId, type (12 types), title, description, metadata, occurredAt | Activity timeline |

### Agent Details

#### 1. Master Agent (Orchestrator)

**Entry point** for the entire pipeline. When started:

1. Loads any uploaded job specification documents
2. Uses Together AI to parse requirements: target roles, required skills, locations, experience level, scoring weights, email tone
3. Auto-creates an email campaign with 3 steps (initial contact, 3-day follow-up, 7-day final follow-up)
4. Generates 10вҖ“15 diverse search queries (Boolean operators, LinkedIn-specific, location-enforced)
5. Dispatches one **Discovery** job per query (staggered with 2-second delays)
6. Stores parsed requirements in `masterAgent.config` for downstream agents

#### 2. Discovery Agent (Web Search + Classification)

Searches the web and classifies results into actionable categories.

**Process:**
1. Executes web search query via SearXNG
2. Uses LLM to classify each result:
   - **Recruitment:** `candidate_profile`, `company_page`, `job_listing`, `irrelevant`
   - **Sales:** `decision_maker`, `company_page`, `team_page`, `irrelevant`
3. Routes by classification:
   - **LinkedIn profiles** вҶ’ saves contact вҶ’ dispatches **Document Agent**
   - **Non-LinkedIn candidates/decision-makers** вҶ’ saves contact вҶ’ dispatches **Enrichment Agent**
   - **Team pages** вҶ’ scrapes page вҶ’ extracts team members вҶ’ creates contacts вҶ’ dispatches **Enrichment** for each
   - **Company pages** вҶ’ saves company with enrichment data
   - **Job listings** вҶ’ saves company with job metadata

#### 3. Document Agent (Scrape & Parse)

Extracts structured data from documents and URLs.

**Handles:**
- LinkedIn profiles (scrape via Crawl4AI вҶ’ extract name, title, skills, experience, education)
- CVs/resumes (PDF/DOCX parsing вҶ’ structured profile extraction)
- Job specifications (parse into target role, required skills, experience level)

**After processing:** dispatches **Enrichment Agent** to continue the pipeline.

#### 4. Enrichment Agent (Deep Multi-Source Research)

The most data-intensive agent. Performs parallel research across multiple sources.

**Phase 1 вҖ” Parallel Source Scraping:**

| Source (Recruitment) | Source (Sales) |
|---------------------|---------------|
| LinkedIn profile | LinkedIn profile |
| GitHub (repos, stars, languages) | Company team page |
| Twitter/X | Company news |
| Stack Overflow | Industry publications |
| Blog / Portfolio | Twitter/X |
| Dev community (dev.to, Medium) | Company LinkedIn page |

**Phase 2 вҖ” Deep Company Enrichment (parallel):**
- Company homepage, About page, Careers page
- Company LinkedIn page
- Crunchbase / funding data
- Recent company news
- Glassdoor reviews
- Key person research with email discovery

**Phase 3 вҖ” Email Discovery:**
- Generates common email patterns (first.last@, firstlast@, f.last@, etc.)
- Performs DNS MX lookup on domain
- Verifies patterns via SMTP handshake
- 30-day Redis cache per result

**Phase 4 вҖ” LLM Synthesis:**
- Merges all sources into a unified `CandidateProfile` or `CompanyProfile`
- Reconciles conflicting data across sources
- Calculates `dataCompleteness` score (0вҖ“100)

**Phase 5 вҖ” Quality Gate:**
- If `dataCompleteness < 30%` вҶ’ archives contact (insufficient data)
- If `dataCompleteness >= 30%` вҶ’ dispatches **Scoring Agent**

#### 5. Scoring Agent (Weighted Evaluation)

Evaluates each contact against the pipeline's requirements.

**Recruitment Scoring Dimensions:**
| Dimension | Description |
|-----------|------------|
| Skills Match | Required vs. actual skills overlap |
| Experience | Years and relevance of experience |
| Location | Proximity to target locations |
| Education | Degree level and relevance |
| Company Background | Quality and relevance of previous employers |

**Sales Scoring Dimensions:**
| Dimension | Description |
|-----------|------------|
| Authority | Decision-making power (C-level, VP, Director) |
| Company Fit | Size, industry, tech stack alignment |
| Accessibility | Availability of direct contact info |
| Relevance | Role relevance to product/service |
| Engagement | Online activity and responsiveness signals |

**Process:**
1. Loads contact data, company data, and master agent requirements
2. Uses Together AI to score 0вҖ“100 with per-dimension breakdown
3. If `score >= threshold` (default 70) вҶ’ updates status to `scored` вҶ’ dispatches **Outreach Agent**
4. If `score < threshold` вҶ’ updates status to `rejected`
5. Logs CRM activity with score details

#### 6. Outreach Agent (Personalized Email)

Generates and queues personalized email campaigns.

**Process:**
1. Loads contact profile, company data, and master agent config
2. Selects best available email sending account (by priority and quota)
3. Uses **Claude AI** to generate a personalized email:
   - References specific skills, projects, or achievements from enrichment data
   - Adapts tone (professional/casual/formal/friendly) per config
   - Includes mandatory content (e.g., Calendly link) per email rules
   - Body limited to <150 words
4. Enqueues email via email queue system
5. Creates or ensures CRM deal exists for this contact
6. Schedules next campaign step (e.g., 3-day follow-up, 7-day final)

**Multi-step campaigns:** The agent calls itself for subsequent steps, each with escalating urgency and different angles.

#### 7. Reply Agent (Classification & Auto-Response)

Classifies incoming email replies and takes automated action.

**Classifications & Actions:**

| Classification | Action |
|---------------|--------|
| `interested` | Schedule interview вҶ’ dispatch **Action Agent** вҶ’ move deal to "Replied" stage |
| `objection` | Send empathetic auto-response вҶ’ keep deal active |
| `not_now` | Schedule re-contact in 60 days вҶ’ dispatch **Outreach Agent** |
| `out_of_office` | Parse return date вҶ’ schedule re-contact after return |
| `unsubscribe` | Archive contact and campaign enrollment |
| `bounce` | Clear invalid email вҶ’ re-dispatch **Enrichment Agent** to find new email |
| `other` | Send generic acknowledgment |

**For general inbound emails (not replies to outreach):**
- Classifies as: inquiry, application, partnership, introduction, support_request, spam
- Auto-responds (except spam)
- Routes to appropriate CRM stage

#### 8. Action Agent (Interview Scheduling)

Executes milestone actions for qualified contacts.

**For `schedule_interview`:**
1. Generates comprehensive candidate report via Together AI (executive summary, skill match analysis, suggested interview questions, hire recommendation)
2. Schedules interview for next business day at 10:00 UTC
3. Creates interview record in database
4. Sends calendar invite email to candidate
5. Sends candidate report to tenant owner
6. Updates contact status to `interview_scheduled`

#### 9. Email Listener Agent (IMAP/POP3 Polling)

Polls configured email inboxes for new messages.

**Process:**
1. Connects to mail server (IMAP via ImapFlow or POP3 via node-pop3)
2. Fetches emails newer than `lastSeenUid`
3. For each email, attempts to match to outbound message using 4 strategies:
   - `In-Reply-To` header matches stored `messageId`
   - `References` header contains stored `messageId`
   - `From` email matches a contact's email (finds most recent sent email)
   - Subject line `Re: X` matching (weak fallback)
4. **Matched replies:** Creates reply record linked to `emailsSent` вҶ’ dispatches **Mailbox Agent** + **Reply Agent**
5. **Unmatched inbound:** Creates new contact if needed вҶ’ creates reply with `isInbound=true` вҶ’ dispatches **Mailbox Agent** + **Reply Agent**
6. Updates `lastSeenUid` for incremental polling

**Polling schedule:** Configurable per listener (default: 60 seconds), managed as BullMQ repeatable jobs.

### Mailbox Agent (Threading & CRM Integration)

Manages email conversation threads with CRM deal linking.

**Thread Email Action:**
1. Finds or creates email thread by contact + normalized subject
2. Updates thread counters (lastMessageAt, messageCount)
3. Links email to thread
4. Ensures CRM deal exists and links to thread
5. Uses LLM to analyze email: extract key info (priority, suggested action, meeting mentions, budget signals)
6. Auto-moves deal stage based on email context
7. Updates thread priority and status

**Additional actions:** Thread summarization, bulk operations (archive, mark spam, unsubscribe), digest generation.

### Email Queue Flow

1. **Enqueue** вҖ” Outreach Agent inserts into `emailQueue` table (status: `queued`) and pushes ID to Redis batch list
2. **Process** вҖ” Email Send Worker runs every 10 seconds, pops up to 10 items from Redis batch list
3. **Send** вҖ” For each queued email:
   - Select best email account (highest priority with available quota)
   - Check daily/hourly quota limits
   - Send via SMTP (Nodemailer)
   - Inject tracking pixel into HTML body
   - Update `emailQueue` status to `sent`
   - Insert into `emailsSent` table
   - Increment quota counters in Redis
   - Log CRM activity (`email_sent`)
   - Emit real-time event
   - Dispatch to **Mailbox Agent** for threading

### Rate Limiting & Quotas

| Level | Limit | Tracking |
|-------|-------|----------|
| Per-account daily | Configurable (default: 500) | Redis counter with daily TTL |
| Per-account hourly | Configurable (default: 50) | Redis counter with hourly TTL |
| SMTP tool rate | 50 emails/sender/day | In-tool rate limiting |

### Warmup Schedule

New email accounts follow a warmup ramp:

| Days Active | Daily Limit |
|-------------|-------------|
| 0вҖ“3 | 5 emails/day |
| 4вҖ“7 | 15 emails/day |
| 8вҖ“14 | 30 emails/day |
| 15вҖ“21 | 50 emails/day |
| 22вҖ“30 | 100 emails/day |
| 30+ | Full configured quota |

### Tracking Pixel

Every HTML email gets an invisible tracking pixel injected:

```html
<img src="{PUBLIC_API_URL}/track/open/{trackingId}" width="1" height="1" style="display:none" />
```

The `/track/open/:trackingId` endpoint returns a transparent 1x1 GIF and asynchronously updates `emailsSent.openedAt`. No authentication required (must work in email clients).

3. **Dashboard receives** events via WebSocket Manager singleton вҶ’ updates Zustand real-time store вҶ’ triggers React Query invalidation

## 14. Security & Multi-Tenancy

### Row-Level Security (RLS)

PostgreSQL RLS enforces tenant isolation at the database level:

```sql
-- Session variable set per request
SET app.current_tenant_id = '{tenant_uuid}';

-- Every query is automatically filtered
CREATE POLICY tenant_isolation ON contacts
  USING (tenant_id = current_setting('app.current_tenant_id')::uuid);
```

All 25 tables have RLS policies for SELECT, INSERT, UPDATE, and DELETE. Tables without direct `tenant_id` columns use transitive checks via foreign keys (e.g., `campaign_steps` checks via `campaigns.tenant_id`).

### Authentication Flow

```
Register/Login вҶ’ JWT access token (15 min) + refresh token (7 days, Redis)
       в”Ӯ
       в–ј
  Every API request вҶ’ Authorization: Bearer {accessToken}
       в”Ӯ
       в–ј
  Token expired? вҶ’ POST /api/auth/refresh вҶ’ new access + refresh tokens
       в”Ӯ
       в–ј
  Refresh expired? вҶ’ Redirect to login
```

### Credential Encryption

Stored email passwords (SMTP, IMAP) are encrypted with AES using `EMAIL_ENCRYPTION_KEY` before database storage. Decrypted at runtime only when needed for email operations.

### Request Validation

All API endpoints validate request bodies with Zod schemas at the Fastify level, rejecting malformed requests before they reach business logic.
